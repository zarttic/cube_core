这套系统解决什么问题

Cube 把 ARD 或观测数据从“对象存储中的文件”变成“有身份、有版本、有质量结果、可入库、可发布的空间数据”。它不会替你决定业务数据是什么，也不会替载入方补造缺失的元数据。

三个角色和分工

角色负责什么不应该做什么
载入方准备文件、checksum、空间时间元数据、波段/变量清单，交付 manifest不要把缺失字段留给前端猜，也不要交付本地路径
操作员对账、选择范围、发起运行、处理失败、触发质检、按波段入库不要直接改数据库状态或重复提交同一运行
平台管理员维护认证、OpenGauss、Ray、MinIO、配置和发布权限不要把凭据写入业务配置表或文档截图

第一次跑一批数据的最短路径

1

准备运行时

检查依赖和 .cube_web.env，启动 API 和前端。

2

验证源对象

在 MinIO stat 源文件，确认 s3 URI、大小和 checksum。

3

导入批次

POST schema import；收到成功响应后，用批次接口对账 Dataset 和 Scene。

4

剖分和观察

页面选择 Scene、格网和层级，记下 partition_run_id 与 task_id。

5

质检、入库、发布

确认 pass 或获批 warn，再按波段入库，最后按权限发布 Dataset 版本。

!

“页面打开了”“接口返回 202”“任务完成了”分别只证明不同层次的事情。完整成功必须能串起 load_batch_id、partition_run_id、output_version、quality_run_id、ingest_run_id 和 publication_id。
