提交前必须先确认

  • LoadBatch 已成功导入，并能查询到 Dataset、Scene、资产和 bands。
  • 每个被选 Dataset 的源对象已在 MinIO stat 成功。
  • 本次运行的 Dataset、Scene 和 BandUnit 不重复；一条 Scene 只能在同一 partition run 中选择一次。
  • 同一 partition run 的所有 Dataset 必须使用同一个格网类型和层级。不同产品可以在不同 partition run 使用不同格网。

页面操作：从批次到运行

1

打开分析就绪数据剖分

管理员进入剖分页后，先选择 optical、radar、product 或 carbon。页面数据来自 GET /v1/partition/load-batches 和对应的 scenes 接口。

2

刷新并打开批次

用批次 ID、数据类型或关键词过滤。打开批次详情，检查 Dataset 数、Scene 数和波段数，确认没有漏选或选错数据。

3

选择 Dataset 和 Scene

选择一个或多个 Dataset，再在每个 Dataset 下选择 Scene。若需要局部运行，可继续指定 band_unit_ids，但不能提交空数组。

4

选择格网和层级

页面用格网类型推导 partition_method：geohash/mgrs 是 logical，isea4h 是 entity。不要把两者当成两个独立下拉框。

5

预览覆盖，再提交

先确认覆盖图和数据范围，再提交。成功响应会返回 partition_run_id、task_id、status、data_type 和 operation。

实际运行请求的结构

{
  "partition_run_id": "RUN_20260719_EAST_001",
  "source_batch_ids": ["LOAD_20260719_001"],
  "selection_source": "load_batch",
  "datasets": [{
    "dataset_id": "GF1-EAST-2020",
    "scene_ids": ["GF1-EAST-20200710"],
    "band_unit_ids": ["GF1-EAST-20200710-B01", "GF1-EAST-20200710-B02"],
    "partition": {"grid_type":"geohash", "requested_grid_level":5}
  }]
}

接口是 POST /v1/partition/runs，响应状态码为 202。前端的草稿接口会先保存选择，再标记草稿 submitted；真正的生产运行仍以 partition run 为准。

格网契约

grid_type派生方式有效层级典型用途
geohashlogical1–12经纬度层次格网
mgrslogical0–5标准 UTM/UPS MGRS
isea4hentity0–15全球六边形实体瓦片
!

s2、tile_matrix、plane_grid 是历史格网，不属于当前生产入口。不要为了绕过校验把它们写进请求。

固定参数与资源控制

参数生产值含义
cover_modeintersect覆盖与资产相交的格网
time_granularityday时间桶按天
max_cells_per_asset0不设上限；不能静默截断或降级
ray_parallelism运行时配置决定 Ray 并发规模，按集群容量设置

冒烟任务可以显式设置小的正数 max_cells_per_asset、ISEA4H level 1、单景和 ray_parallelism 2。冒烟结果只能验证链路，不能当作生产完整覆盖结果。

监控和状态

任务接口：GET /v1/partition/tasks/{task_id}。列表接口：GET /v1/partition/tasks，可按 status、data_type 和 keyword 过滤。

状态你应该做什么
queued等待执行；检查 Ray 地址和 worker 资源，不要重复提交。
running等待或查看 worker 日志；不要再次点击提交。
completed记录 output_version，下一步触发质检。
failed打开详情读取错误；只有最新失败任务允许 retry。
cancelled确认取消原因；需要继续时使用允许的重试入口。

取消和重试

GET  /v1/partition/tasks/{task_id}
POST /v1/partition/tasks/{task_id}/cancel
POST /v1/partition/tasks/{task_id}/retry

重试沿用原 partition_run_id 的业务身份，并只重排符合条件的失败数据单元。不要复制一个新 batch_id 来“重试”，否则会丢失原批次的历史关系。
