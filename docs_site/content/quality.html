什么时候可以质检

剖分任务完成并产生 current output 后，才有可质检对象。页面“质量运行记录”展示的是剖分集合及其 Dataset 质量状态，不是单纯的任务日志。

!

质检完成不会自动创建入库任务。质检是门禁，入库仍需要操作人员在数据管理页明确选择波段。

从页面触发质检

  1. 在质量页找到 partition_run_id，打开详情。
  2. 确认每个 Dataset 有 current_output_version，并查看当前质量状态。
  3. 点击请求质检。页面调用 POST /v1/partition/runs/{partition_run_id}/quality，服务会为等待质检的 Dataset 创建质量运行。
  4. 等待 quality run 到达终态，再打开结果和错误明细。

如果返回 409 partition batch has no completed band units awaiting quality，说明没有完成且等待质检的波段，不要重复点击；先回到剖分任务确认是否真的完成。

质量状态与入库门禁

状态含义入库资格下一动作
pending尚未开始无等待或触发运行
running规则执行中无等待 worker
pass必选规则全部通过可以进入待入库集合
warn只有可选规则告警默认不可以按策略批准 warn 后再入库
fail必选规则产生发现不可以按错误恢复并重质检
error规则、对象读取或执行异常不可以恢复依赖并重试

必选检查项

公共必选规则包括：index_schema、output_count_consistency、output_reference_integrity、grid_method_agreement、cell_bbox_validity、time_bucket_consistency、asset_readability、window_bounds 和 declared_metadata_defects。

产品专用规则还包括：光学的 asset_crs、optical_band_contract；雷达的 radar_band_contract；信息产品的 product_band_contract；碳卫星的 carbon_schema、carbon_coordinates、carbon_xco2_range、carbon_quality_flags、carbon_observation_duplicates 和 carbon_footprints。

如何阅读失败

先看为什么典型处理
rule_code确定哪一类契约失败打开规则目录对照说明
error_code确定具体错误，如 missing_crs、invalid_cog_uri按错误码决定修复层
source_asset_id / band_code定位到具体源资产或波段不要无差别重跑整批
output_id / field定位输出索引中的问题位置修复输出或使用失败重试

导出错误明细

页面分页只是浏览，不代表只有当前页的错误。完整导出使用：

GET /v1/quality/records/{quality_run_id}/errors/export?format=csv
GET /v1/quality/records/{quality_run_id}/errors/export?format=json

导出接口是流式返回，不受列表页 page/page_size 限制。把导出文件与 quality_run_id 一起归档。

失败后的正确顺序

  1. 先判断是源对象/manifest 错误、剖分输出错误，还是 Ray/MinIO/数据库运行时错误。
  2. 源对象问题回到载入方修复并重新交付；不要在质检层伪造观测值或足迹。
  3. 输出问题在原 partition run 中重试失败数据单元。
  4. 服务问题恢复依赖后，在同一 current output 上重新质检。
  5. 重新质检后记录新的 quality_sequence；历史质量运行保留，不能被覆盖。
