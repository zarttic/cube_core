导入到底做了什么

导入接口只登记 LoadBatch、Dataset、Scene、资产和波段，不会自动启动剖分。它把外部载入系统交付的内容变成 Cube 能长期追踪的业务对象。导入成功后，才可以在剖分页面选择数据。

先按四层组织数据

LoadBatch一次载入动作
Dataset长期管理主记录
Scene一次观测/场景
Asset + Band文件和业务数据单元
Output剖分版本

不要把所有资产平铺在顶层，也不要用旧的 batch_id、顶层 data_type、assets 或 observations 代替 Dataset 结构。生产导入契约要求顶层 datasets 非空。

字段怎么填写

字段必须填写规则
load_batch_id是由载入系统生成的稳定 ID；校验失败后用同一 ID 重试。
dataset_id是长期稳定的业务身份，不要每次重传都随机生成。
scene_id是同一 Dataset 内唯一，后续剖分和入库按景追踪。
data_type是只能是 optical、radar、product、carbon。
checksum是64 位小写 SHA-256；先对本地文件计算，再与 MinIO 对象核对。
bands是非空；band_code 是身份，band_name 只用于展示。

四种数据类型的差异

类型源文件band_type额外注意
opticalCOGspectralCOG 必须有 bbox、CRS 和可读波段
radarCOGpolarization典型 VV、VH、HH、HV
productCOGvariable变量含义由产品定义，不由前端猜测
carbonNetCDF/HDF5 原始文件variable只能使用 source_uri；不得传 cog_uri

提交前检查 MinIO 对象

from minio import Minio

client = Minio("10.3.100.179:9000", access_key="<key>", secret_key="<secret>", secure=False)
stat = client.stat_object("cube", "cube/source/optocal/GF1-EAST-20200710.tif")
print(stat.size, stat.etag)

生产脚本应在发送 manifest 前完成 stat，并在对象不存在、大小不符或 checksum 不符时停止。不要把 access key 和 secret 放进 Ray task 参数。

发送请求与判断结果

curl -X POST http://localhost:50039/v1/partition/schemas/import \
  -H 'Content-Type: application/json' \
  --data-binary @load-batch.json
  • 2xx：批次登记成功；记录返回的批次身份，下一步查询 load-batches。
  • 422：领域契约错误，修正原 manifest 后使用相同 load_batch_id 重试。
  • 401/403：认证或角色问题。schema import 在认证开启时仍是公开导入入口，但网关可能有额外限制。
  • 5xx/503：后端依赖或领域存储未配置；先查服务日志和 OpenGauss，不要反复提交。
