# ARD 数据载入到剖分入库与质检业务流程说明

更新时间：2026-07-13

当前契约补充：运行时数据库是 OpenGauss（使用 PostgreSQL 兼容 DSN），对象存储是 MinIO，分布式执行使用 Ray。生产剖分入口使用 `run`；`demo` 只作旧客户端兼容。Web 生产格网为 `s2`、`tile_matrix`、`isea4h`，以及仅支持逻辑方式的实验性 `plane_grid`；`mgrs` 不再出现在生产页面。

本文档不是技术设计说明，也不是接口清单。它面向第一次接触系统的业务人员、实施人员和新同事，目的是讲清楚一批 ARD 数据从进入系统，到完成剖分、入库和质检，中间到底经历了哪些业务环节，每个环节是谁在做、系统在做什么、最后产出什么结果。

## 1. 先用一句话说明这套流程

这套系统做的事情，可以简单理解为：

```text
把 ARD 系统交付的一批遥感数据
整理成可追踪的批次
按照选定格网做剖分
把剖分结果落到共享存储和数据库
再做一轮质量检查
最后把可查、可追溯、可复核的结果交给页面和下游系统
```

如果只从业务角度看，它解决的是三个问题：

1. 一批数据是否已经正式进入平台管理。
2. 这批数据是否已经完成剖分并可用于后续服务。
3. 这批数据的结果是否可靠，能不能被查询、回看和质检追踪。

## 2. 先认识几个业务角色

为了后面容易理解，先把流程里常见的三个角色说清楚。

### 2.1 ARD 数据提供方

这个角色负责把原始处理后的数据交给平台。它不一定直接做剖分，但必须把数据和对应说明准备完整。

它最重要的责任不是“把文件传过来”，而是“把这批数据说清楚”。系统必须知道：

- 这批数据是什么类型。
- 这些数据存放在哪里。
- 每条数据的采集时间是什么。
- 每条数据覆盖的空间范围是什么。
- 每条数据属于什么传感器、产品族、波段或变量。

### 2.2 平台操作人员

这个角色在 Web 页面或通过接口发起正式运行。对他来说，最关心的不是算法细节，而是：

- 这批数据是否已经登记成批次。
- 我要不要现在运行。
- 运行完了没有。
- 如果失败了，应该重跑哪一步。
- 质检结果是否通过。

### 2.3 平台执行系统

这部分是系统自己完成的后台工作。它负责把“我要处理这批数据”真正变成“处理完并留下结果”的完整链路，包括：

- 组织批次
- 执行剖分
- 写入存储
- 执行质检
- 保留过程记录

## 3. 从业务上看，整条链路分成六个模块

为了避免一上来就掉进技术细节，整条流程可以按六个业务模块理解。

1. 数据准备模块：ARD 系统把一批数据和说明交付出来。
2. 批次载入模块：平台把这批数据登记成可管理的业务批次。
3. 运行发起模块：操作人员正式提交剖分任务。
4. 剖分生产模块：系统完成格网剖分，生成可用结果。
5. 入库落库模块：系统把结果写入共享存储和数据库。
6. 质检闭环模块：系统检查结果质量，并把检查结果变成可查询记录。

后面按这六个模块一个一个讲。

## 4. 模块一：数据准备

这是整条链路的起点，也是最容易被低估的一步。

### 4.1 这一步业务上到底在做什么

业务上，这一步不是“传文件”，而是“交付一批可被系统正式接收的数据”。如果只给一堆文件而没有配套说明，系统虽然有一些兼容手段，但那只是兜底，不适合生产。

系统真正需要的是一份清晰的数据交付单。无论它叫 schema、manifest，还是别的名称，本质上都在回答同一件事：

“这一批数据有哪些，每一条数据是什么，时间在哪，空间在哪，后续应该按什么方式处理。”

### 4.2 这一模块的输入是什么

从业务视角看，输入通常包括两部分：

- 数据文件本身。
- 描述这批数据的清单。

对于光学、雷达、信息产品这类栅格数据，清单里通常会有：

- 这批数据的批次标识。
- 每个资产的来源位置。
- 每个资产的时间。
- 每个资产的空间四角范围。
- 每个资产的波段、极化或变量说明。

对于碳卫星这类观测类数据，清单更像观测记录表，重点会落在：

- 观测来源位置。
- 观测时间。
- 观测足迹或点位。
- 观测值及质量标记。

### 4.3 这一模块为什么重要

因为后面所有事情都建立在“这批数据被准确描述”之上。

如果这一层不清楚，后面会出现很典型的业务问题：

- 批次里到底包含哪些数据说不清。
- 运行失败后，不知道是哪个资产有问题。
- 重试时没法精准选择资产。
- 质检结论无法和具体数据一一对应。

### 4.4 这一步做完后，业务上得到什么结果

得到的是一批“可被平台接收的数据候选”。这时它还没有开始剖分，但已经具备了进入平台管理的条件。

## 5. 模块二：批次载入

这一模块的关键词不是“运行”，而是“登记”。

### 5.1 这一步业务上到底在做什么

平台把上一模块交付的数据，正式登记成一个业务批次。登记之后，这批数据不再只是外部文件，而是平台内部一个可跟踪、可查询、可重跑的对象。

你可以把它理解成给这批数据建立了一张“业务档案卡”。

### 5.2 为什么需要“批次”这个概念

因为平台面对的不是单个文件，而是一整批要一起管理的数据。业务上很多动作都是围绕批次展开的：

- 查看这一批现在是什么状态。
- 确认这一批已经跑过几次。
- 找出这一批哪些资产成功，哪些失败。
- 在失败后重试整个批次，或者只重试其中一部分。

没有批次概念，后续的管理和追踪会非常混乱。

### 5.3 这一步系统在后台会做什么

业务上可以简单理解成三件事：

1. 接收这批数据的定义。
2. 记录这批数据包含哪些资产或观测。
3. 给这批数据生成可持续跟踪的批次记录。

这时系统还没有真正开始做剖分，但已经知道“后面要处理谁”。

### 5.4 这一步结束后，页面上能看到什么

理论上你应该能看到：

- 这批数据已经存在。
- 它属于什么数据类型。
- 它包含多少资产或观测。
- 它还没有开始，或者正等待运行。

这就是“数据进平台”和“数据开始处理”之间的分界线。

## 6. 模块三：运行发起

这是业务人员真正按下“开始处理”的那一步。

### 6.1 这一步业务上到底在做什么

操作人员基于一个已登记的批次，发起正式运行。运行发起的含义不是“帮我记一下”，而是“从现在开始，把这批数据送进剖分生产链路”。

在业务上，这一步往往还会带一些运行参数，例如：

- 选哪种格网。
- 用什么层级。
- 是做逻辑剖分还是实体剖分。
- 是否启用分布式并行。

这些参数决定的是“怎么跑”，不是“跑不跑”。

### 6.2 为什么系统要区分“批次”和“运行”

因为一批数据可能不止跑一次。

典型情况包括：

- 第一次运行失败，要重试。
- 运行参数调整后，要重新跑。
- 只针对失败资产再跑一遍。

所以平台通常会保留“这批数据是什么”和“这次具体怎么跑”的分离。前者是业务批次，后者是执行尝试。

### 6.3 这一步结束后，业务上得到什么结果

这时会形成一条明确的执行任务。它至少意味着：

- 平台承认这次运行已经被正式提交。
- 系统会开始处理这批数据。
- 后面所有执行状态都会围绕这次提交进行更新。

从业务视角看，这一步是从“静态管理”进入“动态处理”的起点。

## 7. 模块四：剖分生产

这是整条链路里最核心的生产阶段。

### 7.1 这一步业务上到底在做什么

简单说，就是把原始的 ARD 数据按选定规则切成平台真正可用的格网结果。

业务人员不需要记住内部函数，只要记住它的目标：

- 不是简单搬运文件。
- 不是简单登记路径。
- 而是把原始数据变成按空间和时间组织好的可服务结果。

### 7.2 逻辑剖分和实体剖分有什么业务区别

这是最容易问到的问题之一。

逻辑剖分更像是“建立索引关系”。系统关心的是：

- 这个资产落在哪些格网。
- 每个格网对应哪个时间桶。
- 每个格网该从原始影像的哪个窗口读取。

实体剖分更像是“真正切出结果瓦片”。系统关心的是：

- 这个格网对应的实体瓦片要不要生成。
- 生成后的瓦片存放在哪里。
- 后续服务是不是可以直接使用这个瓦片。

所以从业务效果看：

- 逻辑剖分更偏“形成索引能力”。
- 实体剖分更偏“形成可直接分发的瓦片成果”。

### 7.3 系统在这一阶段通常会做什么

从业务角度，可以把后台动作理解成下面几步：

1. 读取这一批数据的清单。
2. 确认每个数据项的时间、空间和类型信息。
3. 根据格网规则计算每条数据要落到哪些空间单元。
4. 按时间维度把结果归到对应时间桶。
5. 生成这一批剖分结果的中间成果。

如果是栅格类逻辑剖分，结果更像一份“索引明细表”。

如果是实体剖分，结果更像“瓦片成果 + 对应索引关系”。

### 7.4 为什么这一步常常要依赖共享存储和分布式资源

因为正式业务数据量通常不小，而且很多任务要在多台机器上并行跑。

业务上只需要记住一条原则：

一旦进入分布式运行，就不能假设所有节点都能看到同一台机器的本地文件。

这就是为什么生产链路强调：

- 源数据尽量走共享对象存储。
- 处理过程允许 worker 本地缓存。
- 结果最终还要回到共享可访问的位置。

### 7.5 这一阶段结束后，业务上得到什么结果

这时系统已经拿到了“剖分成果”。它可能表现为：

- 一批格网索引行。
- 一批实体瓦片及其索引行。
- 一批观测类剖分结果。

但到这里还不算全流程结束，因为这些结果还需要正式落库和归档。

## 8. 模块五：入库落库

这一模块最容易和上一模块混在一起，但业务意义并不一样。

### 8.1 这一步业务上到底在做什么

剖分生产完成后，系统需要把成果从“刚算出来”变成“平台正式持有”。这个动作就是入库落库。

业务上它至少包含两层含义：

- 结果文件或瓦片要落到共享存储。
- 结果的描述信息要写进数据库。

只有这两件事都完成，后续查询、回看、调度和下游使用才有可靠基础。

### 8.2 为什么不能只保留文件，不写数据库

因为平台后续要解决的不只是“文件存在”，还包括：

- 这份结果属于哪个批次。
- 这份结果对应哪个空间单元和时间桶。
- 这份结果来自哪次运行。
- 这份结果是哪个版本。

这些问题不靠文件目录本身解决，必须有数据库里的结构化记录。

### 8.3 为什么不能只写数据库，不保存共享成果

因为很多业务能力后面真的要拿到数据本体：

- 读取瓦片。
- 下载成果。
- 做后续质量复核。
- 支持回放、重算或外部服务调用。

只有记录，没有实际成果文件，平台只是“知道它存在”，却拿不到它。

### 8.4 这里有一个关键业务事实

当前生产主流程里，很多数据类型在正式 `run` 中是自动入库的。

这意味着业务上不要把“提交剖分”和“确认入库”理解成两段必然分开的流程。对当前系统来说，更接近真实情况的理解是：

```text
正式运行
  = 剖分处理
  + 结果落库
  + 结果登记
```

也就是说，系统不是先停在半路等人点第二个按钮，而是尽量一次跑完整条主链路。

### 8.5 这一步结束后，业务上得到什么结果

到这里，一批数据已经不只是“跑过了”，而是“正式进入平台成果体系”。

你应该能基于它做这些事情：

- 查这一批对应哪些结果。
- 追踪某个结果来自哪次运行。
- 从共享存储拿到相应成果。
- 对结果做后续质检和服务发布。

## 9. 模块六：质检闭环

质检不是附属动作，它是主流程的最后一道业务关口。

### 9.1 这一步业务上到底在做什么

系统会对刚完成的剖分结果做一轮检查，回答一个业务问题：

“这批结果看起来是否完整、合理、可交付。”

它不是在重新跑剖分，而是在检查剖分后的结果是否站得住。

### 9.2 质检通常关心什么

从业务上看，质检一般会围绕这些问题展开：

- 结果文件或索引记录是否真的生成了。
- 结果是否为空。
- 必要字段是否齐全。
- 时间、空间和统计信息是否符合预期。
- 是否出现明显异常或告警。

不同数据类型检查重点会略有不同，但业务目的相同：确认结果可用性。

### 9.3 为什么质检结果还要单独存档

因为质检不是一次性的口头结论，而是一份需要长期保留的业务记录。

后续常见需求包括：

- 页面查看最新报告。
- 查询历史报告。
- 导出报告做汇报或归档。
- 对比同一批数据不同运行的质量结果。

如果每次都去现场重新扫运行目录，不仅慢，而且不利于审计和追溯。

### 9.4 这一步结束后，业务上得到什么结果

这时平台最终形成的是一份完整闭环：

- 数据已经进入平台。
- 剖分已经执行。
- 成果已经入库。
- 质量状态已经可查。

也就是说，这批数据从“被接收”走到了“可交付、可追溯、可复核”。

## 10. 把六个模块串起来看一遍

如果要给新人讲整条业务线，可以直接按下面这段话说：

第一步，ARD 系统准备好一批数据，并把每条数据的时间、空间和类型说明清楚。第二步，平台把这批数据登记成一个正式批次，让它进入统一管理。第三步，操作人员发起运行，告诉系统这批数据现在可以开始处理。第四步，系统按照选定格网完成剖分生产，生成空间时间组织后的成果。第五步，系统把成果写入共享存储和数据库，让它成为平台正式持有的结果。第六步，系统再做一轮质检，把质量结论保存下来，供页面查询和后续复核。

理解了这六步，基本就理解了这套系统的业务主线。

## 11. 业务现场最常问的 QA

### Q1：为什么不能只给一个目录，让系统自己识别？

可以兜底识别，但不适合作为正式生产流程。业务上真正需要的是“可追踪的数据交付”，不是“系统猜一猜这是什么”。只给目录会带来两个直接问题：

- 交付边界不清，事后说不清这一批到底包含哪些数据。
- 出问题时很难快速定位是哪条数据描述不完整。

所以正式流程里，推荐一定要带清单。

### Q2：为什么平台要先建批次，不能一上来直接跑？

因为业务上要管理的是“一批数据的生命周期”，不是“一次瞬时计算”。建批次以后，后面才能说清楚：

- 这批数据是谁交付的。
- 这批数据何时进入平台。
- 它跑过几次。
- 哪次成功，哪次失败。
- 后续重试针对的是哪一批。

### Q3：为什么正式运行后，很多时候不需要再人工确认入库？

因为当前主流程的设计目标，就是尽量把剖分、落库和登记串成一次完整执行。业务上把它理解成“一键跑完整条生产链”更接近真实情况，而不是“先算一半，再等人工补最后一步”。

### Q4：为什么系统强调共享存储，而不是直接用本地路径？

因为在分布式场景下，本地路径只对当前节点可见。业务上如果希望一批任务能在多节点稳定运行，就必须把源数据和成果放在所有节点都能访问的位置。共享对象存储就是这个统一位置。

### Q5：质检通过是不是就代表数据一定完全正确？

不是。质检通过代表的是“系统按当前规则没有发现明显问题，结果达到了当前可交付标准”。它是上线前的重要关口，但不是无限上纲的绝对保证。业务上仍然要根据场景决定是否做人工抽查或专题复核。

### Q6：如果运行失败，第一反应应该是什么？

先不要急着整批重跑，先判断失败发生在哪个模块：

- 如果是数据描述不完整，先回到“数据准备”修清单。
- 如果是执行资源或并行问题，重点看“剖分生产”阶段。
- 如果是结果没落全，重点看“入库落库”阶段。
- 如果是报告异常，重点看“质检闭环”阶段。

只有先分清失败属于哪一层，重试才会有针对性。

### Q7：为什么质检结果还要保留历史，而不是只看最新一次？

因为业务上很多问题都依赖对比：

- 同一批数据这次为什么通过，上次为什么告警。
- 同一类数据在不同时间是否有稳定性变化。
- 某次生产调整后，质量是否变好或变差。

没有历史，就只能看到一个静态结果，看不到过程。

## 12. 给新人的最终理解方式

不要把这套系统理解成“一个剖分程序”，那样会越看越碎。更准确的理解应该是：

它是一条数据生产流水线。

前半段解决“把数据接进来并说清楚”。

中间段解决“把数据变成按格网组织的正式成果”。

后半段解决“把成果落稳、记清，并给出质量结论”。

如果你在工作中要判断当前卡在哪，只要先问自己一句：

“现在卡的是数据准备、批次载入、运行发起、剖分生产、入库落库，还是质检闭环？”

多数问题，走到这一步就已经找对方向了。
