# 剖分入库与 AOI 读取流程

更新时间：2026-04-23  
适用项目：`cube_split`  
关联项目：`cube_encoder`

---

## 1. 文档目标

本文档记录当前已经跑通的工程链路，而不是泛化方案：

- 景级遥感数据如何完成逻辑剖分与入库
- `rs_cube_cell_fact` 如何作为空间索引主表使用
- AOI 如何解析成多个 `space_code`
- 如何基于 `value_ref_uri#window=...` 从 MinIO 回源读取
- 如何把多个格网单元、多个波段的数据合成为一个 GeoTIFF

一句话概括：

`景级 TIF -> COG -> 格网覆盖 -> 生成索引行 -> 入 PostgreSQL -> AOI 解析为 space_code[] -> 查询索引 -> 按 window 读取 MinIO -> 合成 tif`

## 3. 剖分入库流程

## 3.1 输入

输入通常是景级原始波段文件，例如：

- `LC08_L2SP_120030_20260204_20260217_02_T1_SR_B2.TIF`
- `LC08_L2SP_120030_20260204_20260217_02_T1_SR_B3.TIF`
- `LC08_L2SP_120030_20260204_20260217_02_T1_SR_B4.TIF`

当前样例数据位于：

- `../cube_split/data/landsat8`

## 3.2 COG 标准化

剖分前先把输入 TIF 转为 COG，默认放到：

- `data/cog/partition_input`

这一步由 `ray_logical_partition_job.py` 调用 `convert_assets_to_cog(...)` 完成，目的有两点：

- 统一后续按 window 随机读取的访问路径
- 避免对原始 TIF 做重复非优化读取

## 3.3 格网覆盖

每个 COG 资产根据其 footprint 执行格网覆盖，核心参数包括：

- `grid_type`
- `grid_level`
- `cover_mode`
- `time_granularity`

当前实测参数：

- `grid_type = geohash`
- `grid_level = 7`
- `cover_mode = intersect`
- `time_granularity = day`

覆盖后，每个资产会拆成多条“资产-格网单元”记录。

## 3.4 索引行产出

剖分作业输出 `index_rows.jsonl`，每一行代表：

- 某个 `scene_id`
- 某个 `band`
- 在某个 `space_code`
- 某个时间桶 `time_bucket`
- 对应原始 COG 中的一个窗口

当前实测样例：

```json
{
  "scene_id": "LC08_L2SP_120030_20260204_20260217_02_T1",
  "band": "qa_pixel",
  "asset_path": "/home/lyjdev/projects/cube_split/data/cog/partition_input/LC08_L2SP_120030_20260204_20260217_02_T1_QA_PIXEL_cog.tif",
  "acq_time": "2026-02-04T00:00:00Z",
  "grid_type": "geohash",
  "grid_level": 7,
  "space_code": "5e15c",
  "st_code": "gh:7:5e15c:20260204:v1",
  "time_bucket": "20260204",
  "window_col_off": 867,
  "window_row_off": 0,
  "window_width": 3711,
  "window_height": 866
}
```

关键点：

- `space_code` 是空间索引键
- `time_bucket` 是时间索引键
- `window_*` 是数据回源读取的窗口定位信息

## 3.5 元数据入库

入库脚本 `ray_ingest_job.py` 负责把剖分结果写入 PostgreSQL 和对象存储。

实际落表有 3 张：

1. `rs_raw_scene_asset`
2. `rs_cube_cell_fact`
3. `rs_ingest_job`

其中对查询最关键的是 `rs_cube_cell_fact`。

---

## 4. 核心表说明

## 4.1 `rs_raw_scene_asset`

作用：

- 保存景级资产元数据
- 保存原始 COG 路径
- 提供血缘与审计能力

主键约束：

- `UNIQUE(scene_id, band, version)`

## 4.2 `rs_cube_cell_fact`

作用：

- 作为“无景化”的主查询表
- 以格网单元为主键组织数据
- 对外不直接按 `scene_id` 查询

业务唯一键：

- `grid_type + grid_level + space_code + time_bucket + band + cube_version`

关键字段：

- `grid_type`
- `grid_level`
- `space_code`
- `time_bucket`
- `band`
- `st_code`
- `cell_min_lon/cell_min_lat/cell_max_lon/cell_max_lat`
- `value_ref_uri`
- `source_scene_count`
- `provenance_json`

其中：

- `space_code` 是空间索引
- `time_bucket` 是时间索引
- `value_ref_uri` 是数据读取入口

## 4.3 `rs_ingest_job`

作用：

- 保存任务状态、参数、统计信息

主要用于：

- 跑批监控
- 失败重试
- 批次回溯

## 5. `value_ref_uri` 的含义

当前实现中，`rs_cube_cell_fact.value_ref_uri` 不是直接存“小切片文件路径”，而是存：

- MinIO 中的原始 COG 对象路径
- 外加该格网单元对应的窗口位置

形式如下：

```text
s3://cube/cube/raw/dataset=landsat8/sensor=L8/acq_date=2026/02/04/scene_id=LC08_L2SP_120030_20260204_20260217_02_T1/version=v1/LC08_L2SP_120030_20260204_20260217_02_T1_SR_B4_cog.tif#window=867,0,3711,866
```

拆开后可得到：

- 对象位置：`s3://...SR_B4_cog.tif`
- 读取窗口：`col_off=867,row_off=0,width=3711,height=866`

这意味着当前系统采用的是：

- 索引驱动
- 按需窗口读取
- 不预先把每个格网物理切成独立文件

## 6. AOI 读取流程

## 6.1 基本逻辑

读取不是“按景扫描”，而是：

1. 输入 AOI
2. 通过格网覆盖得到 `space_code[]`
3. 用 `space_code[] + time_bucket + band` 查询 `rs_cube_cell_fact`
4. 拿到每条记录的 `value_ref_uri`
5. 从 MinIO 对应 COG 中按 window 读取
6. 将多个 cell、多个 band 合成输出结果

所以读取分成两层：

- 索引层读取：读 PostgreSQL 的 `rs_cube_cell_fact`
- 数据层读取：读 MinIO 上 COG 的窗口数据

## 6.2 单 cell 读取

单 cell 测试示例：

- `grid_type = geohash`
- `grid_level = 7`
- `space_code = 5e15c`
- `time_bucket = 20260204`
- `band in (sr_b2, sr_b3, sr_b4)`

读取结果：

- 直接按 3 条 `value_ref_uri` 回源读取 `sr_b2/sr_b3/sr_b4`
- 合成为一个三波段 GeoTIFF

输出文件：

- `/home/lyjdev/projects/cube_encoder/.tmp/merged_5e15c_20260204_sr_b2_b3_b4.tif`

## 6.3 AOI 多 cell 读取

本次 AOI 测试输入：

```text
bbox = [120.8, 44.0, 122.2, 44.6]
grid_type = geohash
level = 7
cover_mode = intersect
time_bucket = 20260204
bands = [sr_b2, sr_b3, sr_b4]
```

AOI 解析得到 7 个 `space_code`：

```text
5e144, 5e154, 5e15c, 5e164, 5e16c, 5e3dc, 5e3e4
```

其中在当前日期上真正命中 3 波段记录的有 5 个：

```text
5e15c, 5e164, 5e16c, 5e3dc, 5e3e4
```

读取与拼接过程：

1. 对每个有效 `space_code` 查询 `rs_cube_cell_fact`
2. 每个 cell 取 `sr_b2/sr_b3/sr_b4` 三条记录
3. 解析 `value_ref_uri#window=...`
4. 从 MinIO 按窗口读取对应像元
5. 根据窗口的地理变换把多个 cell 镶嵌到统一画布
6. 输出三波段 GeoTIFF

最终输出文件：

- `/home/lyjdev/projects/cube_encoder/.tmp/aoi_multi_spacecode_20260204_sr_b2_b3_b4.tif`

结果摘要：

- 有效格网数：`5`
- 输出尺寸：`6714 x 3443`
- 坐标系：`EPSG:32651`
- 波段：`sr_b2 / sr_b3 / sr_b4`
- 文件大小：约 `60M`

像元统计：

- `sr_b2`: `min=0, max=60287, mean=5638.15`
- `sr_b3`: `min=0, max=46422, mean=6518.97`
- `sr_b4`: `min=0, max=41431, mean=7471.39`

这个测试证明：

- AOI 可以稳定解析为多个 `space_code`
- 读取是按格网空间索引驱动的
- 数据层确实是按 COG window 回源，不是整景全量读取

## 8. 关键结论

1. 格网剖分结果本身就是空间索引，`space_code` 不只是编码字段，而是主读取入口。
2. `rs_cube_cell_fact` 是当前读取主表，查询时必须优先走 `space_code + time_bucket + band`。
3. `value_ref_uri#window=` 是连接“索引层”和“数据层”的关键桥梁。
4. 当前链路已经跑通：
   - 景级数据剖分入库
   - 单 cell 三波段读取
   - AOI 多 cell 多波段回源并合成 tif
5. 当前输出仍然是“读取后再拼接”的结果文件，不是把多 cell 预先物化成一个长期保存的 cube tile。

## 9. 后续建议

建议下一步工程化补齐以下能力：

1. 把 AOI 多格网读取固化成脚本或 API，例如：
   - `read_aoi_rgb.py`
   - `/v1/cube/read-aoi-rgb`
2. 为读取链路增加分页和批量限制，防止 AOI 过大时一次扫描过多 `space_code`。
3. 增加读取结果元数据输出：
   - 实际命中的 `space_code`
   - 缺失 `space_code`
   - 每个 band 的命中条数
4. 根据查询模式决定是否引入预物化层：
   - 热区缓存
   - month 聚合层
   - 长期稳定的多波段 tile 产物
