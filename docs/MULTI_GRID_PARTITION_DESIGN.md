# 多格网剖分流程设计

## 目标

同一数据单元需要支持以下三种格网的独立剖分流程：

- Geohash
- MGRS
- ISEA4H

上述名称以当前 `cube_encoder` 和 `cube_split` 的生产实现为准。生产执行器当前支持集合为
`geohash | mgrs | isea4h`，本功能不引入新的格网引擎。

每个剖分批次只能使用一种格网类型和一个格网层级。批次可以包含多个数据集、景和波段，但批次中的所有数据必须使用相同的格网配置。

## 数据选择粒度

数据管理页面按照以下层级选择待剖分数据：

`数据集 -> 景 -> 波段`

用户可以选择整个数据集、部分景，或者景中的部分波段。已经完成对应格网全流程的数据单元需要显示完成标记并禁止再次选择。

## 状态粒度

状态以 `数据单元 + 格网类型` 为最小记录粒度，格网层级记录为该格网当前采用的层级。数据集和景的状态由下级数据单元聚合计算，波段直接展示具体状态标签。

建议新增状态表：

```text
partition_data_unit_grid_status
- dataset_id
- scene_id
- band_unit_id
- grid_type
- grid_level
- partition_run_id
- partition_status
- quality_status
- ingest_status
- output_version
- error_message
- created_at
- updated_at
```

唯一约束为：

```text
(band_unit_id, grid_type)
```

## 载入批次可见性

载入批次只承担首次剖分入口。

- 只完成部分数据时，批次继续保留。
- 已走完入库流程的数据单元显示完成标记并禁止选择。
- 只有批次内所有数据单元都完成入库后，载入批次才从待剖分列表中消失。
- 数据单元是否完成以入库状态为准，单独完成剖分或质检不代表流程完成。

## 后续格网剖分

完成首次流程后，用户从数据管理页面查看数据集、景和波段在三种格网下的状态，并选择尚未完成的格网类型组成新的剖分批次。

新批次必须统一格网类型和格网层级。批次完成后更新对应数据单元的格网状态，不覆盖该数据单元其他格网类型的结果。

## 失败与重试

- 剖分失败在原 `partition_run_id` 中重试，只重新处理失败数据单元。
- 质检失败按错误来源处理：元数据问题直接修正后重检；产物问题在原批次局部重建；源数据问题退回载入系统修正。
- 入库失败只重试失败数据单元，已经入库成功的数据保持完成状态。
- 重试不会创建新的业务批次，但需要记录每个数据单元的尝试次数和独立输出版本。
- 执行器按波段隔离任务；同景波段共享同一源资产定位，但每个 `band_unit_id` 独立产出状态、错误和重试记录。
