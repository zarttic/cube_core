# 多格网剖分流程设计

## 目标

同一数据单元需要支持以下三种格网的独立剖分流程：

- Geohash
- MGRS
- ISEA4H

上述名称以当前 `cube_encoder` 和 `cube_split` 的生产实现为准。生产执行器当前支持集合为
`geohash | mgrs | isea4h`，本功能不引入新的格网引擎。

一个 `partition_run_id` 是一个逻辑剖分批次和一个顶层任务。它可以包含多个选择项；每个
选择项绑定自己的载入批次、数据集、景/波段及格网类型和层级。执行器在同一顶层任务内按
选择项独立执行，因此同一景从不同载入批次重新载入后，可以在同一剖分批次中分别按
Geohash、MGRS 或 ISEA4H 配置剖分。

剖分页面按产品类型维护四个相互独立的待提交上下文：光学、雷达、碳卫星和信息产品。
一个上下文可以选择多个同产品载入批次，并在一次提交中合并生成一个 `partition_run_id`；
不同产品上下文不能互相显示或被隐式合并提交。提交成功只清空当前产品的临时选择，不改变
来源载入批次的业务状态。

## 数据选择粒度

数据管理页面按照以下层级选择待剖分数据：

`数据集 -> 景 -> 波段`

用户可以选择整个数据集、部分景，或者景中的部分波段。已经完成对应格网全流程的数据单元需要显示完成标记并禁止再次选择。

同一产品页选择的多个载入批次属于一次待提交组装，不复用任何来源 `load_batch_id` 作为
剖分批次标识。服务端接受提交后生成新的 `partition_run_id`，供后续质检和入库按合并后的
剖分批次处理。

## 状态粒度

状态以 `数据单元 + 格网类型 + 格网层级` 为最小记录粒度。数据集和景的状态由下级数据
单元聚合计算，波段直接展示具体状态标签。

建议新增状态表：

```text
partition_data_unit_grid_status
- dataset_id
- scene_id
- band_unit_id
- grid_type
- grid_level
- partition_run_id
- partition_status
- quality_status
- ingest_status
- output_version
- error_message
- created_at
- updated_at
```

唯一约束为：

```text
(band_unit_id, grid_type, grid_level)
```

## 载入批次可见性

载入批次是每次正式载入动作的统一入口，既包括子系统导入，也包括数据集重新载入。二者都
使用 `load_batches` 和 `load_batch_scenes`；区别只通过来源元数据和血缘字段表达。未确认的
用户编辑仅保存在 `partition_drafts`，草稿不在剖分批次页面伪装为载入批次。

- 只完成部分数据时，批次继续保留。
- 已走完入库流程的数据单元显示完成标记并禁止选择。
- 只有批次内所有数据单元都完成入库后，载入批次才从待剖分列表中消失。
- 数据单元是否完成以入库状态为准，单独完成剖分或质检不代表流程完成。

## 后续格网剖分

完成首次流程后，用户从数据管理页面查看数据集、景和波段在三种格网下的状态，确认重新
载入后生成新的正式载入批次，再在剖分页面选择一个或多个正式批次。

一个剖分批次中的每个选择项保留自己的格网类型和层级。批次完成后更新对应数据单元的
精确格网状态，不覆盖该数据单元其他格网类型或层级的结果。

## 失败与重试

- 剖分失败在原 `partition_run_id` 中重试，只重新处理失败数据单元。
- 质检失败按错误来源处理：元数据问题直接修正后重检；产物问题在原批次局部重建；源数据问题退回载入系统修正。
- 入库失败只重试失败数据单元，已经入库成功的数据保持完成状态。
- 重试不会创建新的业务批次，但需要记录每个数据单元的尝试次数和独立输出版本。
- 执行器按波段隔离任务；同景波段共享同一源资产定位，但每个 `band_unit_id` 独立产出状态、错误和重试记录。
